# How To: Add a Neo4j Graph RAG Tool to askMilo

Companion to [README_HOWTO-addToolToAskMilo.md](README_HOWTO-addToolToAskMilo.md). This covers the Neo4j-specific steps for creating a RAG search tool backed by a Neo4j Docker container with native vector search.

**For the general setup** (provider.json, .ini config, registration, deployment), see the main guide.

---

## Overview

The Neo4j RAG pipeline uses `indexTextGraphForMilo` to:

1. **Start** a Neo4j Docker container (one per indexed collection)
2. **Chunk** source documents using Claude (Anthropic API)
3. **Embed** chunks using Voyage AI (1024 dimensions by default)
4. **Store** chunks as Neo4j nodes with vector embeddings and a vector index
5. **Scaffold** a provider directory with search CLI and provider.json

Unlike the SQLite version, the data lives in a Docker container's persistent volumes, not in a `.db` file. The provider directory contains the search script, `provider.json`, `chunks.jsonl`, and an `assets/` directory with a portable data tarball.

---

## Prerequisites

- `indexTextGraphForMilo` CLI tool at: `qbookSuperTool/system/code/cli/lib.d/index-text-graph-for-milo/indexTextGraphForMilo.js`
- **Docker** installed and in PATH
- A Voyage AI API key (for embedding generation and query-time embedding)
- An Anthropic API key (for chunking — the chunker calls Claude)
- Text files (`.md`, `.txt`, or similar) in a source directory

## Step 1: Start the Neo4j Container

```bash
# Manual start (first-time setup without pre-built assets)
node /path/to/index-text-graph-for-milo/indexTextGraphForMilo.js \
  -start \
  --name=YourProviderName \
  --voyageApiKey=pa-YOUR_VOYAGE_API_KEY

# Automatic start (via provider.json — uses pre-built assets from -indexText)
node ../index-text-graph-for-milo/indexTextGraphForMilo.js -start --assetsDir=./assets
```

### What `-start` Does (First Run — No Assets)

1. Scans for available ports starting from 7700 (Bolt + HTTP pair)
2. Creates directory structure at `dataStores/instanceSpecific/index-text-graph-for-milo/`
3. Creates Neo4j data directories (data, logs, plugins, import)
4. Writes an instanceConfig `.ini` with ports, container name, password, API key
5. Runs `docker run` with the picked ports and `--restart unless-stopped`
6. Waits for Neo4j to become reachable (polls Bolt port, max 60s)
7. Creates the vector index for chunk embeddings
8. **Auto-generates** the search `.ini` config (see Step 3)

### What `-start` Does (First Run — With `--assetsDir`)

When `-start` finds `manifest.json` + `neo4j-data.tar.gz` in the assets directory (produced by `-indexText` Step 3.5), it performs a **portable cold start**:

1. Reads providerName, password, and embedding config from `manifest.json`
2. Scans for available ports (same as above)
3. Extracts the pre-built Neo4j data from the tarball instead of creating empty directories
4. Creates the Docker container with the extracted data already in place
5. **Skips vector index creation** (the index is already in the extracted data)
6. Auto-generates the search `.ini` config

This is the standard deployment path — the `provider.json` generated by `-indexText` includes `"startup": "node ../index-text-graph-for-milo/indexTextGraphForMilo.js -start --assetsDir=./assets"`, so the provider carries its own data and bootstraps automatically on any server.

### What `-start` Does (Subsequent Runs)

It's **idempotent**: checks if the container is already running, starts it if stopped, does nothing if running. Safe to run on every boot. Also ensures the search `.ini` exists on every run.

### Other Container Commands

```bash
# Check container status
indexTextGraphForMilo -status --name=YourProviderName

# Stop the container
indexTextGraphForMilo -stop --name=YourProviderName

# Destroy everything and start fresh (requires --confirm)
indexTextGraphForMilo -forceInit --name=YourProviderName --confirm
```

## Step 2: Index the Documents

```bash
node /path/to/index-text-graph-for-milo/indexTextGraphForMilo.js \
  -indexText \
  --sourcePath="/path/to/your/text/files/*.md" \
  --providerName=YourProviderName \
  --outDirPath=/path/to/output/provider-directory \
  --chunkStrategy="your chunking instructions"
```

### Parameters

| Parameter | Required | Description |
|-----------|----------|-------------|
| `-indexText` | Yes | Action flag (single hyphen, qtools convention) |
| `--sourcePath` | Yes | Path or glob to source text files |
| `--providerName` | Yes | CamelCase name (e.g., `CareerStoryGraph`). Must match the `--name` used in `-start`. |
| `--outDirPath` | Yes | Where to write the output provider directory |
| `--chunkStrategy` | No | Natural language chunking instructions, or a predefined shortcut. Defaults to paragraph-level. |

### What It Produces

```
your-provider-directory/
  yourProviderNameSearch.js    # Search CLI (generated from neo4jSearchTemplate.js)
  provider.json                # Tool definition for askMilo registry (includes startup command)
  chunks.jsonl                 # Human-readable source chunks (archive)
  assets/
    neo4j-data.tar.gz          # Portable Neo4j data tarball (for cold starts on any server)
    manifest.json              # Provider config (name, password, embedding settings)
```

Note: **No `.db` file** — the data lives in the Neo4j container's Docker volumes at `dataStores/instanceSpecific/index-text-graph-for-milo/neo4j-{name}/`. The `assets/` tarball is a portable copy used by `-start --assetsDir` to bootstrap on new servers without needing to re-run `-indexText`.

### Re-indexing

Running `-indexText` again on an existing container **replaces** all data (DETACH DELETE all Chunk nodes, then reloads). An `--append` mode may be added in the future.

## Step 3: The Search .ini Config (Auto-Generated)

**The search `.ini` is auto-generated by `-start`.** You do not need to create it manually. Every time `-start` runs (including via provider.json startup), it writes or updates the search config.

The search script's module name determines the `.ini` filename:

- Script: `careerStoryGraphSearch.js` → Config: `careerStoryGraphSearch.ini`

**Location:** `qbookSuperTool/system/configs/instanceSpecific/qbook/careerStoryGraphSearch.ini` (local) or `configs/careerStoryGraphSearch.ini` (server, flat layout)

### What Gets Generated

```ini
[careerStoryGraphSearch]
neo4jBoltUri=bolt://localhost:7704
neo4jUser=neo4j
neo4jPassword=GENERATED_PASSWORD
voyageApiKey=YOUR_VOYAGE_API_KEY
embeddingModel=voyage-4
embeddingDimensions=1024
topK=10
sourceLabel=CareerStoryGraph
```

The port and password come from the instanceConfig at `dataStores/instanceSpecific/index-text-graph-for-milo/index-text-graph-for-milo.ini`. The Voyage API key is passed through from `--voyageApiKey` or the askMilo config.

### Important: Voyage API Key

The auto-generated `.ini` will have an **empty `voyageApiKey`** unless you either:
1. Pass `--voyageApiKey=KEY` to `-start`, or
2. Have it configured in `askMilo.ini`

If the key is missing, searches will fail at query time (can't generate the query embedding). After `-start` runs, check the generated `.ini` and add the key manually if needed.

## The provider.json Startup Feature

The Neo4j provider's `provider.json` includes a `startup` command with `--assetsDir`:

```json
{
    "providerName": "CareerStoryGraph",
    "startup": "node ../index-text-graph-for-milo/indexTextGraphForMilo.js -start --assetsDir=./assets",
    "startupTimeout": 30,
    "tools": [...]
}
```

When askMilo loads its provider registry, it runs this startup command automatically. The `--assetsDir=./assets` tells `-start` to look for pre-built data (the tarball + manifest produced by `-indexText`). Since `-start` is idempotent, this ensures the Docker container is running and populated whenever askMilo is invoked — no separate boot-time script or server-side indexing needed.

The startup command runs with `cwd` set to the provider directory. The `../index-text-graph-for-milo/` relative path resolves because the generated provider lives alongside the indexer tool in `cli/lib.d/`. The `./assets` path resolves to the provider's own `assets/` directory.

## Server Deployment Notes

### Docker on the Server

The server must have Docker installed. The Neo4j container runs as a user-level Docker container (no root required if the user is in the `docker` group).

### First Deploy (Standard — With Pre-Built Assets)

After `deployPrograms educore --actions=configs,code,restart`:

1. **The provider directory** (search script, provider.json, chunks.jsonl, **assets/**) is rsynced to the server
2. **The indexer tool** (index-text-graph-for-milo/) is also rsynced
3. **On first askMilo invocation**, the startup command in provider.json runs `-start --assetsDir=./assets`, which:
   - Reads providerName and password from `assets/manifest.json`
   - Creates the `dataStores/instanceSpecific/` directory on the server
   - Picks available ports
   - **Extracts the pre-built Neo4j data** from `assets/neo4j-data.tar.gz`
   - Starts the Docker container with the extracted data already in place
   - Skips vector index creation (already in the data)
   - Auto-generates the search `.ini` config
4. **The container is fully populated and ready** — no server-side `-indexText` needed

This is the standard workflow. The tarball carries the indexed data from your dev machine to the server automatically.

### Alternative: Indexing on the Server

If you need to index different source documents on the server (or re-index with new data), you can run `-indexText` directly:

```bash
# SSH to server
ssh cfnew

# Index the documents (source files must be on the server)
node /home/educore/system/code/cli/lib.d/index-text-graph-for-milo/indexTextGraphForMilo.js \
  -indexText \
  --sourcePath="/path/to/source/files/*.md" \
  --providerName=YourProviderName \
  --outDirPath=/home/educore/system/code/cli/lib.d/your-provider-directory
```

### Server .ini

The search `.ini` is auto-generated by `-start` on the server. However, the Voyage API key may be empty if not configured. Check `configs/careerStoryGraphSearch.ini` on the server after the first startup and add the key if missing.

## Neo4j Data Model

### Node Schema

```cypher
(:Chunk {
    chunkId: STRING,        // UUID
    text: STRING,           // chunk content
    label: STRING,          // chunk label from chunker
    sourceFile: STRING,     // origin file path
    startLine: INTEGER,     // line number in source
    embedding: LIST<FLOAT>  // Voyage AI embedding vector (1024 dims)
})
```

### Vector Index

```cypher
CREATE VECTOR INDEX chunk_vector_index IF NOT EXISTS
FOR (c:Chunk) ON (c.embedding)
OPTIONS {indexConfig: {
    `vector.dimensions`: 1024,
    `vector.similarity_function`: 'cosine'
}}
```

### Search Query

```cypher
CALL db.index.vector.queryNodes('chunk_vector_index', $limit, $embedding)
YIELD node, score
RETURN node.text AS chunk_text,
       node.sourceFile AS source_file,
       node.label AS label,
       node.startLine AS start_line,
       score
ORDER BY score DESC
```

## Portability

The Neo4j provider is **portable via the assets tarball** (requires Docker on the target). To move to a new server:

1. Deploy code via `deployPrograms` (provider directory with `assets/` + indexer tool)
2. On first askMilo invocation, `-start --assetsDir=./assets` automatically:
   - Creates the Docker container
   - Extracts pre-built data from the tarball
   - Generates the search `.ini` config
3. Verify the Voyage API key in the generated `.ini`

No server-side `-indexText` needed — the data travels with the provider.

The Docker container uses `--restart unless-stopped`, so it survives server reboots. Data is on bind-mounted volumes, so it survives container recreation.

## Comparison: SQLite vs Neo4j RAG Tools

| Concern | SQLite | Neo4j |
|---------|--------|-------|
| **Embedding provider** | OpenAI text-embedding-3-small | Voyage AI voyage-4 |
| **Dimensions** | 1536 | 1024 |
| **Data portability** | Single `.db` file, rsync anywhere | Tarball in `assets/` — auto-extracted by `-start` on first run |
| **Server requirements** | Node.js + native modules | Node.js + Docker |
| **Setup complexity** | Lower (no container management) | Higher (Docker lifecycle) |
| **Startup** | Instant (file-based) | Container boot (~10-30s first time) |
| **Query capability** | Vector similarity only | Vector similarity + graph traversal potential |
| **Best for** | Simple document collections, maximum portability | Collections where graph relationships matter, or when you prefer Voyage AI embeddings |
